# Скилл: Code Review

## Когда применять
Пользователь просит проверить код, найти баги, улучшить структуру, оценить решение.

## Что проверять

### 1. Корректность
- Логические ошибки
- Граничные случаи (пустые массивы, null, нулевые значения)
- Гонки, race conditions

### 2. Безопасность
- Утечки секретов (API-ключи в коде, .env в репо)
- SQL/command injection
- Невалидированный пользовательский ввод
- Открытые CORS, отсутствие auth

### 3. Читаемость
- Понятные имена переменных и функций
- Функции до 50 строк
- Без вложенности глубже 3 уровней
- Комментарии объясняют ПОЧЕМУ, не ЧТО

### 4. Производительность
- N+1 запросы к БД
- Лишние циклы и копирования
- Отсутствие пагинации/лимитов

## Формат ответа

```
## Найдено

### 🔴 Критично (надо чинить)
- [строка X] Проблема — почему критично

### 🟡 Стоит улучшить
- [строка X] Описание

### 🟢 Хорошо
- Что сделано правильно (1-2 пункта)

## Конкретные правки
[блоки кода: было → стало]
```

## Принципы

- Не «всё плохо» и не «всё ок» — конкретика
- Предлагай решение, не только проблему
- Объясняй ПОЧЕМУ это проблема
- Похвали то что хорошо
